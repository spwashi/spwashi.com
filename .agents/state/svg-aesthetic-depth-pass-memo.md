## 2. Aesthetic Diagnosis

**What is already strong.** The site has a real contract waiting to be used. The combination of `spw-shell.css`'s layout variants, `spw-wonder.css`'s SVG-aware lifecycle, `svg-surfaces.css`'s operator-typed node palette, and the resonance system in `spw-wonder.css:745-803` already names every primitive a serious diagrammatic depth pass would need. The `spw-mark.svg` infrastructure shows that the craft guard (`viewBox` discipline, addressable IDs, CSS-token stroke) is achievable in this hand-authored idiom without tooling. `about/website/index.html:747-758` proves the host contract reads fluently when actually applied.

**What currently limits interactive wonder.** The host contract is opt-in but almost no one opts in correctly, and the one route that most owes the user a demonstration of SVG craft — `topics/craft/svg/index.html` — bypasses the contract entirely with bespoke `.craft-story-svg` classes and hardcoded gradient stops. As a result, wonder mechanisms attached to `.spw-svg-surface` (charge states, wonder accents, visited glow) silently miss most inline SVG on the site. Inline diagrams have no addressable nodes, no `data-spw-operator` attributes, and no host id, so the resonance system in `spw-wonder.css:745` cannot make them participate when a user explores related operators elsewhere on the page. The diagrammatic surfaces are alive but isolated from the rest of the cognitive substrate.

**Where responsive behavior is structurally thin.** Three gaps stand out. (a) Tablet (roughly 720–1152px) gets the worst of both worlds: the header has already collapsed to toggle mode, the split-rail layout has not yet activated, and SVG hosts neither shrink progressively nor gain auxiliary context. (b) Wide displays (`≥1152px`) only gain wider gutters; the `data-spw-layout="atlas"` and `split` variants exist but `topics/craft/svg/index.html` uses the default reading width, so a 1600px monitor renders the storytelling SVG at the same scale as a 900px laptop with empty space on either side. (c) Long/tall displays gain neither sectional anchoring (the `.spw-section-handle` is `≤720px` only) nor SVG-aware orientation aids — a long page of multiple SVG hosts has no register of "which surface am I currently inside." The shell already names these conditions; SVG content does not yet vary with them.

**On taste.** The risk in a depth pass is that "more interactive wonder" becomes more animation, denser ornament, or hero inflation. The repo's stance — `diagrammatic warmth + inspectable restraint`, `wonder as relation becoming legible`, "if the same insight survives without the flourish, keep the structure and discard the flourish" (`svg-surface-integration.spw:107`) — should be defended explicitly. Every recommendation below is structural rather than animated.

## 3. Three Recommended Improvements

### Improvement 1 — Extend the manifest to a typed SVG inventory and adopt the host contract on existing inline SVGs

**Why this matters.** The manifest already counts SVGs per route but does not know what any of them *are*. Until each inline `<svg>` declares its kind, host id, scale range, and interaction model, agents cannot help with depth work — they can only grep. And until the canonical host contract (`.spw-svg-surface` / `.spw-svg-figure` / `data-spw-svg-host`) is actually used, the wonder, resonance, and persona systems silently skip most inline SVG. This single gap is what prevents the rest of the depth pass from compounding.

**Repo scope.** Two layers, no build tooling.
- `scripts/site-contracts.mjs:416-422` and `scripts/site-contracts.mjs:451-481` extend the per-route `svg.*` block and the top-level `maps.svgAssets` list to capture, per asset and per inline host: declared `data-spw-svg-kind` (`illustration | addressable-illustration | widget | component`), `data-spw-svg-host` id, `viewBox` aspect ratio, `data-spw-svg-scale` (e.g. `16-256`, `narrow-wide`), `data-spw-svg-motion` (`static | reveal | resonant`), and presence of `<title>` / `<desc>` ids.
- HTML adoption pass on the four routes already opting into `svg-surfaces` (notably `topics/craft/svg/index.html:144-227`), wrapping each inline SVG in `<figure class="spw-svg-figure" data-spw-svg-host="…">` with `<svg class="spw-svg-surface spw-svg-surface--…" data-spw-svg-kind="…">` and replacing hardcoded fills with `currentColor` / `--op-*-color` / existing `--craft-*` tokens.

**Breakpoint strategy.**
- *Mobile.* No new behavior; the host contract is the prerequisite. `spw-svg-surface` already gets a tighter `border-radius` at `≤700px` (`svg-surfaces.css:147`).
- *Tablet.* Manifest fields enable a follow-up to size SVG hosts to a "compact" variant via `data-spw-svg-scale="narrow-wide"`.
- *Wide.* With kind known, a `data-spw-svg-kind="addressable-illustration"` can opt into wider scaling without inflating widget kinds.
- *Long/tall.* Hosts gain a stable id, which a future section-handle variant can target for "you are inside diagram X."

**Interaction / wonder model.** No new interactions. The point is to make existing wonder primitives find their targets. Once `.spw-svg-surface` is in place, the charge/wonder/visited rules in `spw-wonder.css:60-73, 657-673` apply automatically.

**Agentic benefit.** `npm run manifest` becomes the single source for "what SVG surfaces exist, and what kind are they" — exactly the question the agentic-dev-contracts plan was designed to answer. It also lets `scripts/check-site.mjs` warn when an inline SVG omits required host metadata.

**Implementation sketch (no code).** In `parseRouteFile` (`scripts/site-contracts.mjs:321`), add a small inline-SVG pass that captures opening `<figure class="spw-svg-figure"` and `<svg …>` tags, collects the new `data-spw-svg-*` attributes, and emits an `svgHosts: [{ id, kind, scale, motion, hasTitle, hasDesc, viewBox, route }]` array per route. Aggregate into `maps.svgHosts` alongside `maps.svgAssets`. Then walk `topics/craft/svg/index.html` and convert one host at a time; each conversion is reversible and locally inspectable.

**Risk / tradeoff.** The bespoke `.craft-story-svg` styling encodes specific tonal choices (the warm paper gradient, the halo). Converting blindly to `currentColor` flattens that. The conversion has to preserve the warm/paper feel by introducing per-kind tokens (e.g. `--craft-story-paper`, `--craft-story-halo`) rather than collapsing to a single accent. There is also a soft risk of over-typing the manifest with kinds that turn out to be wrong; start with the four taxonomy values from the spw plan and resist adding more.

### Improvement 2 — Tablet-and-up "rail companion" for SVG hosts using the existing `split`/`atlas` layout primitives

**Why this matters.** Today the site has two width modes for SVG: full-width inline within `reading` width, or full-width inline within `wide` width. Tablets get the worst of this — the header has already gone toggle mode but the rail layout has not activated. Wide screens just gain margin. The site already has the right primitive (`data-spw-layout="split"` with `.spw-gutter-rail`) but it is reserved for desktop reading routes. SVG hosts that carry inspectable structure (legends, node lists, "currently revealed" registers) deserve a rail companion as soon as horizontal space exists, which is roughly tablet-portrait and above.

**Repo scope.**
- `public/css/svg-surfaces.css` gains a host-scoped variant: when an SVG host declares `data-spw-svg-companion="rail"`, it participates in a local two-column grid with a sibling `.spw-svg-rail` element rather than depending on the page's `data-spw-layout="split"`.
- Routes adopt the rail companion selectively: `topics/craft/svg/index.html` is the obvious first candidate (legend chips already exist as `.spec-strip` siblings; they could move into a rail at tablet).
- Optionally extend `parseRouteFile` to count `data-spw-svg-companion` usage in the manifest.

**Breakpoint strategy.**
- *Mobile (≤720px).* No companion. Rail content collapses below the SVG as a captioned strip, similar to how `.spw-gutter-rail` collapses today (`spw-shell.css:260-266`).
- *Tablet (~720–1152px).* Companion activates if the SVG's intrinsic aspect ratio leaves room — e.g. `viewBox` aspect < 2.4. Rail width clamped narrower than desktop (e.g. `min(20%, 14rem)`).
- *Wide (≥1152px).* Companion gets full rail (`min(28%, 18rem)`), aligned to a sticky position similar to `spw-shell.css:238-247`.
- *Long/tall.* The rail can sticky-pin within the host's bounding box, so as the user scrolls past one SVG into the next, each SVG's rail handoff feels local rather than global.

**Interaction / wonder model.** Hovering or focusing a node in the SVG (with stable `id` and `data-spw-operator`) updates a register in the rail showing "current node" and "related operators," using the existing resonance system. No new motion. Dismissing focus settles the rail back to its idle state. This is the canonical "reveal layers, correspondences, or transforms; do not rely on perpetual animation" pattern from `svg-surface-integration.spw:104`.

**Agentic benefit.** Adds one declarative attribute (`data-spw-svg-companion`) that the manifest can count and validate. Makes wide-display behavior structural rather than asking individual routes to roll their own.

**Implementation sketch (no code).** Define `.spw-svg-host--with-rail` (or scope via `[data-spw-svg-companion="rail"]`) that becomes a CSS grid at `(min-width: 720px)`. The rail is the next sibling with class `.spw-svg-rail`. Reuse the wonder substrate's `[data-spw-charge]` and `[data-spw-wonder-state]` for rail entries. Initial demonstration uses the `topics/craft/svg/index.html` story SVG: the existing `.spec-strip` of `motifs / illustrator handoff / inspectable marks` becomes rail entries that arm when their corresponding glyph is focused.

**Risk / tradeoff.** Two-column layouts inside an article create reading-flow hazards if the rail grows tall enough to outrun the SVG. Mitigations: clamp rail max-height to the SVG's rendered height with `overflow-y: auto`; collapse rail back to flow on `≤720px`; never put primary content in the rail. Also: do not let the rail become a second navigation system. Reserve it for register-style reveal, not for jumping elsewhere.

### Improvement 3 — Long-display anchoring: an SVG-aware section handle and resonance-pinning across multi-SVG pages

**Why this matters.** A page with several inline SVGs (today's `topics/craft/svg/index.html`, tomorrow's expanded craft and software routes) reads on a long display as a stack of unrelated illustrations because nothing tells the user which surface they are currently inside, and nothing carries a relational signal between surfaces. The `.spw-section-handle` already solves the "where am I" problem on mobile (`spw-chrome.css:1150-1227`); the resonance system already solves "what is related" within a single page. Neither is wired to SVG. A small, careful unification — not animation, not ornament — would make a long-display SVG-rich page feel anchored and resonant rather than scrolling-through-an-album.

**Repo scope.**
- `public/css/spw-chrome.css` `.spw-section-handle` extends to expose the active SVG host id when the visitor's scroll position is inside one. Driven by IntersectionObserver hooked to `[data-spw-svg-host]`.
- `public/css/spw-wonder.css` resonance section gains a parallel for inline SVG: when the user focuses or hovers an SVG node carrying `data-spw-operator="<op>"`, the existing `html[data-spw-resonance-probe="…"]` mechanism pins, and other SVGs on the page with nodes of the same operator gain a soft accent — the same mechanism today handles `.operator-chip` resonance.
- An optional `data-spw-section-handle-desktop="auto"` opts long-form pages into the section handle above 720px as well; today it is mobile-only.

**Breakpoint strategy.**
- *Mobile (≤720px).* Section handle already exists; gains the SVG host name when scrolled into one.
- *Tablet (720–1152px).* Section handle activates if the page declares `data-spw-section-handle-desktop="auto"`; this is the breakpoint with the most pages-with-inline-SVG-but-no-rail.
- *Wide (≥1152px).* Optional. On wide, the rail companion from Improvement 2 may render section handle redundant; one or the other should win per-page, not both.
- *Long/tall.* The whole point. The handle stays useful as the user scrolls past several SVGs; resonance pinning means hovering an operator in SVG #1 leaves a faint accent on SVG #3 below the fold, so when the user scrolls there, the relationship is already legible.

**Interaction / wonder model.** Three lightweight states, no new motion. (a) Idle: handle is hidden. (b) Inside an SVG host: handle shows host id and the host's primary operator. (c) Pinned: a long-press or keyboard activation pins the current operator's resonance across the page; resonance settles back when dismissed. The resonance pinning piggybacks on `html[data-spw-resonance-probe]` (`spw-wonder.css:795-803`); no new global state.

**Agentic benefit.** All anchoring uses existing data attributes (`data-spw-svg-host`, `data-spw-operator`, `data-spw-resonance-probe`). The manifest can validate that pages with multiple inline SVGs declare host ids; the section handle then becomes self-explanatory rather than requiring per-route JS.

**Implementation sketch (no code).** Reuse the existing IntersectionObserver pattern that drives `.spw-section-handle` today. When the observed elements include `[data-spw-svg-host]`, emit the host's id into `.spw-section-handle__label`. For resonance pinning, extend the existing `html[data-spw-resonance-probe="…"]` toggle: a long-press or a "pin operator" affordance on a focused SVG node sets the probe and shows a tiny "unpin" chip. Demonstration on `topics/craft/svg/index.html`: scrolling between the storytelling SVG, the showcase cards, and the next-step section keeps the user oriented; hovering a `^frame`-typed node in any host pins resonance across the page.

**Risk / tradeoff.** The section handle was designed as a calm, mobile-only affordance; promoting it to desktop risks visual clutter, especially when combined with the rail from Improvement 2. The mitigation is pages opt in (`data-spw-section-handle-desktop="auto"`) rather than blanket activation, and rail-bearing pages skip the handle by default. Pinned resonance can also become noise if every operator the user touches stays accented; require an explicit pin gesture rather than implicit hover-to-pin, and always provide an obvious dismissal path.

## 4. Adoption Path

**Priority order.**

1. **Improvement 1 first.** Without manifest typing and the host contract actually in use, Improvements 2 and 3 have nothing to attach to. This is also the smallest reversible change and the one that lowers agent friction the most, which matches the agentic-dev-contracts goal in `.agents/plans/agentic-dev-contracts/PLAN.md:7`.
2. **Improvement 2 second.** Activates a meaningfully different tablet/wide experience using primitives that already exist in the shell.
3. **Improvement 3 last.** Builds on the host ids established in Improvement 1 and the rail discipline from Improvement 2.

**How to roll out without breakage.**

- *Improvement 1.* Add manifest fields as additive (no validator failure if absent). Convert `topics/craft/svg/index.html` one SVG at a time, preserving its warm/paper palette by introducing per-kind tokens before changing fills. Leave `about/website/index.html:747-758` untouched as the reference shape.
- *Improvement 2.* Ship `data-spw-svg-companion="rail"` as opt-in. The first adopter is the storytelling SVG on `topics/craft/svg/index.html` because it already has natural rail content (`.spec-strip` chips at lines 134-140). Skip routes that today have no rail content; do not invent it.
- *Improvement 3.* Ship `data-spw-section-handle-desktop="auto"` as opt-in, default off. Resonance pinning can land independently and is broadly safe because it reuses the existing probe attribute.

**Concrete worked example (existing route).** `topics/craft/svg/index.html` is the cleanest demonstration site:

- *Today.* 5 inline SVGs with bespoke classes and hardcoded hex stops (lines 145-227). The page declares `svg-surfaces` but doesn't use the contract. No tablet adaptation. No anchoring on long display.
- *After Improvement 1.* Each SVG sits inside `<figure class="spw-svg-figure" data-spw-svg-host="craft-story|craft-glyph|craft-divider|craft-creature">`, the SVG element is `<svg class="spw-svg-surface spw-svg-surface--craft" data-spw-svg-kind="addressable-illustration" data-spw-svg-scale="narrow-wide" …>`, fills route through `currentColor` plus per-kind craft tokens. `npm run manifest` lists each host with its kind and scale.
- *After Improvement 2.* The `craft-story` host declares `data-spw-svg-companion="rail"`; the existing `.spec-strip` chips become rail entries at `≥720px`. Hovering a glyph inside the SVG arms the corresponding rail entry, and vice versa. Mobile collapses to today's behavior.
- *After Improvement 3.* The page opts into `data-spw-section-handle-desktop="auto"`. As the visitor scrolls between the storytelling host and the three showcase hosts, the section handle names the current host. Hovering a `^frame`-typed glyph in any host pins resonance, accenting matching nodes in the other hosts.

Each step is reversible and locally inspectable.

## 5. Open Questions

- **Per-kind palette tokens vs. operator tokens.** The repo already has `--op-*-color` (semantic operator) and `--craft-*` (route-flavored). Should `data-spw-svg-kind="addressable-illustration"` always inherit operator tokens, or should route-flavored tokens override per-host? This is taste, not architecture.
- **Where the host contract should be enforced.** Should `scripts/check-site.mjs` warn when a route declares `data-spw-features~="svg-surfaces"` but contains inline `<svg>` outside `.spw-svg-surface`? Warning is cheap; making it an error would break `topics/craft/svg/index.html` immediately.
- **Tablet rail or no tablet rail.** A rail companion at 720–1152px solves the "tablet emptiness" problem but compresses the SVG. Worth confirming on actual tablet hardware before committing the breakpoint.
- **Resonance pinning gesture.** Long-press is intuitive on touch but conflicts with iOS context menu. A small explicit "pin" chip avoids the conflict but adds chrome. Which feels calmer in practice?
- **Section handle on desktop.** The mobile section handle is welcome partly *because* it is mobile-only. Promoting it to desktop on long pages risks feeling redundant next to scroll position. Worth prototyping before broad adoption.
- **Whether `.spw-svg-surface` should ungate.** Today the rules in `svg-surfaces.css` are gated by `[data-spw-features~="svg-surfaces"]`. Once the host contract is real, should the gate move to the host element instead so the contract works without route-level opt-in? Affects backwards behavior on every existing inline `<svg>`.
- **External illustrations under `public/images/illustrations/`.** The svg-surface-integration plan reserves this directory but it is empty. Should the depth pass commission a small set of authored illustrations, or should it stay confined to inline diagrams until illustrator handoff is real?
- **Who owns the rail content register.** If rail entries arm/disarm based on SVG node focus, and resonance pinning sets a global probe, those two systems can race. A small precedence rule is needed (e.g. "pinned operator wins over hover; node focus wins over pinned operator only inside its own host"), but that is a design call, not a discoverable fact.

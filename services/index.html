Services

I build systems that teach their own structure.

A system I build has three layers: a reading layer, a material layer, and a semantic contract layer. The reading layer is prose and structure — content that teaches itself without a user manual. The material layer is how the page resists and yields: CSS as physics, components with state, transitions that have destinations. The semantic contract layer is what makes the system extensible — every element has a declared role, a touch channel, a memory policy. Other tools can read it. Other engineers can reason about it.

More than a decade of frontend systems, developer tooling, content platforms, and product architecture. The work is visible in the grammar surface, spw-workbench, and every Wednesday at the RPG table.

What I Do

Frontend systems

Architecture from the reading layer down. The goal is a page that feels trustworthy enough to think in — consistent behavior, legible structure, states that announce themselves. Semantic HTML, accessible components, progressive enhancement, local-first tooling that works offline and doesn't ask for an account.

Language and grammar tooling

Grammar design, parser systems, component contract systems. The work here is making structure visible — so it can be argued with, extended, or handed to someone else. Pretext.js is the clearest example: it measures, computes, and sets properties. The layout becomes a transition target rather than something that reflows unexpectedly.

Product architecture

Scope that's honest about what it excludes. API contracts, technical priorities, and staffing decisions that survive a quarter without documentation. I work at staff level — the conversation is about tradeoffs, not just implementation. Interactive curricula and learning surfaces are a specific strength.

Play and facilitation

RPG Wednesday is proof of collaborative intelligence under uncertainty. The table doesn't lie about how someone handles ambiguity, defers to others, or improvises under pressure. Game design and facilitation are applied learning science — the same structures show up in onboarding, documentation, and interface design.

How I Work

The work is public and social. The best collaborations come from shared context — seeing how I think, watching how the tools behave, playing at the same table.

RPG Wednesday

A recurring Wednesday tabletop session. Watching how someone runs a table or plays a character tells you something real about how they think, collaborate under uncertainty, and treat other people's ideas.

If you play tabletop RPGs and want to see how I work with a group, this is the most honest context available.

Pretext.js

A measurement-driven text layout library. No DOM reflow. Reads body copy, writes CSS custom properties, lets layouts breathe correctly at any size. The code is the calling card.

If you're an engineer evaluating the work, read the source. The quality of the library tells you more than a portfolio page.

spw-workbench

The public parser, runtime, and tooling work. Where the grammar research and systems thinking are most visible. Where I push on deep algorithmic problems.

Long-horizon work, visible in public. The kind of project that takes years and produces durable things.

This site

The site itself is a working system: semantic HTML as a conceptual ledger, CSS as material physics, JS as readable state transitions. Every component has a contract. The interaction model is intentional.

Inspect the source. The architecture is the portfolio.
